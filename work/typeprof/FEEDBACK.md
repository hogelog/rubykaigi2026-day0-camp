# themes/typeprof.md への学習者フィードバック

Ruby 4.0.2 + `typeprof 0.31.1` + `rbs 3.10.0` で入門・中級・上級(Ruby Scripts モード)を
一通り触った結果、`themes/typeprof.md` に対して感じたことのまとめ。
「合宿参加者目線で読みやすくするための改善提案」として残す。

## よかった点

- 「触って分かると嬉しいこと」の **4 項目(RBS 基本文法 / 得意な所・諦める所 / sig/ の作法 / Steep との役割分担)**
  が、実験のメニューとしてそのまま機能する粒度だった。
- 入門・中級・上級の **段差がはっきり** している。入門(小さな script に当てる)で
  RBS を読む体力がついた状態で、中級(推論外れ)の実験をすると「TypeProf は何を見ていて
  何を見ないか」が腑に落ちる。
- 「予想される詰まりどころ」のうち「メタプロが絡むと Any になる」は実物で確認できた
  (`define_method` / `Struct.new` / `Data.define` で**クラスが生えない**)。先に読んで
  おいたことで「これは想定内」と落ち着いて観察に回れた。
- 参考リンクの **mame の RubyKaigi 2025 "Writing Ruby Scripts with TypeProf"** を
  先に見てから触ると、v0.30+ の `#:` アノテーション体験の意義がわかる導線になっていた。

## もう少しあると嬉しかった点

以下は「初見で詰まった」「書いてあれば 15 分節約できた」系の具体フィードバック。

### 1. 「入門」の最初の一歩: 3 秒で走るワンライナーが欲しい

ガイドは「50〜200 行のスクリプトに typeprof をかけて出力 RBS を読む」と書いているが、
**手を動かす前にまず typeprof がどんな出力を吐くかのサンプルが 1 つ欲しい**。
たとえば:

```sh
echo 'def add(a, b) = a + b; add(1, 2)' > /tmp/t.rb
typeprof /tmp/t.rb
# => def add: (Integer, Integer) -> Integer
```

この 3 行で「あ、実引数から型を伝播するのか」と掴める。本格的なスクリプトは
その後で十分。

### 2. 「Array 引数はタプル型で出る」罠を先回りする

実験で一番驚いたのがこれ。`find_even([1, 3, 5])` を呼ぶと TypeProf は
**`Array[Integer]`** ではなく **`[Integer, Integer, Integer]`**(3 要素固定タプル)と
推論する。そのまま RBS にコピペして公開すると、5 要素の Array を渡した瞬間に
型が合わない RBS になる。

ガイドに 1 行あるだけで救われる:

> TypeProf は Array リテラルを固定長タプルとして覚える癖があるので、
> 出力 RBS はそのままではなく `Array[T]` に書き直してから `sig/` に
> コミットすることが多い。

### 3. `puts` が `(*String)` 想定である(= Integer 渡すとエラー)

`typeprof --show-errors` をかけると、**`puts double(3)` のような
`puts(Integer)` が "wrong type of arguments" で弾かれる**。RBS の Kernel#puts が
`(*String)` で定義されているのが原因。`p(Integer)` は通る。

これは「サンプルの末尾に `puts add(1, 2)` と書いた瞬間にエラー」で初見殺し。
ガイドに:

> `--show-errors` を付けると `puts` に String 以外を渡しているだけで
> エラーが出る。動作確認には `p` の方が摩擦が少ない。

くらいのメモがあると、初見の驚きが減る。

### 4. `Struct.new` / `Data.define` は諦められる、を明記

「メタプロで諦める」は一般論として書かれているが、**`Struct.new(:x, :y)` の x / y 読み書きすら
TypeProf 0.31 では追えない** のは事前に知っておきたい重要事項。

`Point = Struct.new(:x, :y) do; def distance; Math.sqrt(x*x + y*y); end; end` を
書くと、`distance` メソッドが **`Object` 直下** に生え、`undefined method: Object#x`
エラーまで出る。**Struct / Data は sig/*.rbs で手書き RBS を用意する前提**、の
ひと言があると設計で迷わない。

### 5. `#:` アノテーションの導入例を 1 段目に入れてほしい

v0.30+ の売りである **「Ruby ファイルの `def` 直前に `#: (...) -> ...` と書くと
TypeProf が宣言として読む」** は、実は**入門レベルで一番体験の差が出る機能**。
mame の 2025 発表を指すリンクはあるが、**ガイド本文に最小例**:

```ruby
#: (Integer, Integer) -> Integer
def add(a, b)
  a + b
end

add(1, "oops") # --show-errors でここが型エラーになる
```

があるだけで、「TypeProf は単なる雛形ジェネレータではなく、型チェッカとしても
使える」という**現行版の立ち位置**が一発で伝わる。

### 6. `#:` アノテーションのキーワード引数は 0.31 では効かない、の注記

`#:` を試した時に **キーワード引数の型指定だけ効かない**癖があった:
- `(String, ?loud: bool) -> String` と書いても、出力は `(String, ?loud: false) -> String`(デフォルト値の**値型**に落ちる)
- 呼び出し側で `loud: "truthy"` を渡しても型エラーが出ない

位置引数と戻り値は効くので、**「キーワード引数が絡む場合は `sig/*.rbs` 外出しで
指定する」**という回避策を添えておくと、初見で「アノテーション壊れてる?」と
沼に入らずに済む。(0.31.1 時点での挙動。上流で修正されるかもしれない)

### 7. `typeprof.conf.jsonc` と `rbs_dir: sig/` の存在を一言

ガイドに `sig/` ディレクトリの話はあるが、**`typeprof --init` で
`typeprof.conf.jsonc` が生える**ことには触れていない。合宿で Rails や gem に
typeprof を当てるときの**最初の一手**が `--init` なので、

> プロジェクト直下で `typeprof --init` を叩くと
> `typeprof.conf.jsonc` が生成され、`rbs_dir: "sig/"` が既定で
> 設定される(LSP モード / エディタ連携でも読まれる)。

の一行があると楽。

### 8. `rbs prototype rb` / `rbs prototype runtime` との使い分けの一文

ガイドには「`rbs prototype runtime` との違いを比べる」課題があるが、**答えの骨子**が
本文に無い。比較してみて分かったのは:

| ツール | 一言で | 使いどころ |
| --- | --- | --- |
| `rbs prototype rb` | AST から `untyped` の雛形を吐く | 新規コードに型を付け始める第一歩 |
| `rbs prototype runtime` | ロード後にリフレクションで見る | 動的に生える gem のメソッドも拾える |
| `typeprof` | 抽象実行で**型を埋める** | 実際に型チェック / 型推論したい時 |

この 3 行が本文にあると、読者が「あ、役割違うんだ」と気付く。

### 9. `--show-errors` を付けると挙動が大きく変わる

これも罠要素。`typeprof file.rb` と `typeprof --show-errors file.rb` で**出力の
意味合いが違う**(後者はエラーを出力、前者は沈黙して RBS だけ吐く)。
「型チェックとしての TypeProf を体験するには `--show-errors` が必要」を明記したい。

## 上級への入り口の設計

- **「TypeProf の内部実装(抽象解釈器)を軽く読む」**課題の具体的な足場として、
  `lib/typeprof/core/ast/method.rb` の `get_rbs_comment_before`(`#:` アノテーション
  のパーサ)と、`lib/typeprof/core/service.rb` の `@rbs_text_nodes` 周り(RBS を
  どう取り込んでいるか)から入ると、**ソースの地理が短時間で掴める**。
- **抽象解釈器の本丸は `lib/typeprof/core/graph/*.rb`**(`box.rb` など)。
  メソッドを呼ぶたびに頂点と辺を増やすデータフロー型の抽象実行で、
  RBS 型をその頂点の制約として流していく形。最初の 1 段落に「どのファイルを
  見るか」の指差しがあると、読者が脳内で地図を描ける。
- gem_rbs_collection の PR 練習については、**既存 PR を 1 つ読む** → **型を足したい
  メソッドを選ぶ** → **`rbs prototype` で雛形** → **手で埋めて `rbs validate`** の
  4 ステップを並べておくと、ハードルがぐっと下がる。

## 道具の限界の明示

本合宿の TypeProf / RBS / Steep は、次の種類の問題は**見えない**:

- **値の範囲**(ゼロ除算、負の添字、オーバーフロー)
- **副作用の有無**(純粋関数かどうか)
- **所有権 / ライフタイム**(どこで pin されているか)
- **並列性 / 共有可能性**(Ractor テーマの `shareable?` は型システムの管轄外)

逆に、**発見できる**のは:
- 「宣言した型 vs 呼び出し引数型」の不一致
- 「宣言した戻り値型 vs 本体の返す型」の不一致
- メソッド未定義(typo や refactor 取りこぼし)

この**役割の線引き**を冒頭に 1 段落置くと、「型を入れれば全部検出できる」幻想が
消えて、合宿中の議論が前に進みやすそう。

## まとめ

`themes/typeprof.md` は既に良い取り組みアイデア集になっている。**入門の最初に
動く 3 行サンプルを置く**、**Array タプル化 / `puts` の `(*String)` / Struct 諦め /
`--show-errors` で動作が変わる**の 4 つの罠を明記する、**`#:` アノテーションの
最小例**を本文に入れる、の 3 点を足すと、初見の参加者が v0.30+ TypeProf を
しっかり体感できるガイドになりそう。
